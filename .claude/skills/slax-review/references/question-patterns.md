# Question Patterns — slax-review

何时用 `AskUserQuestion` 工具问开发者？以下是典型场景。

**记住**：意图不明、有多种实现可能、平台行为差异 —— 都该问，不要瞎猜。

---

## 📐 提问的硬性原则

1. **分轮问** —— 每轮 2-4 个最关键的问题，等回答完再下一轮。不要一次甩 10 个问题。
2. **给选项，不开放式** —— 用 `options` 数组（2-4 个），让用户快选。
3. **附 (Recommended)** —— 自己判断更可能的选项放第一位，label 末尾加 `(Recommended)`。
4. **header 要短** —— ≤ 12 字符的 chip 标签，例如 `"Auth method"`、`"Coroutine"`、`"Compose key"`。
5. **影响结论的优先** —— 不问就给不出准确 review 的问题，先问；纯好奇的可以归到「报告附带」。

---

## 🎯 典型问题场景

### 0️⃣ 上下文与规范确认（阶段 0 专用）

> **场景**：拿不到 PR 需求描述。
- "这次改动的需求/目的是什么？解决什么问题？预期行为变化？"
  - 没选项 —— 让用户自由描述（这是为数不多需要开放式提问的场景，但也建议把开放式答案选项放最后，先给 1-2 个常见类型供快选）
  - 例如选项：`"修 bug"` / `"加新功能"` / `"重构 / 优化"` / `"其他（详述）"`

> **场景**：项目没找到任何规范文件。
- "项目目录下没找到 CLAUDE.md / AGENTS.md / CONTRIBUTING.md / 编码规范文档 —— 接下来怎么处理？"
  - 选项：
    1. `"现在生成一份基于 KMP+Compose 行业最佳实践 (Recommended)"`
    2. `"暂不生成，本次按通用标准 review"`
    3. `"跳过，我自己另外补"`

> **场景**：改动涉及 API contract / 序列化模型。
- "这次改动了 Ktor 的 path / @Serializable model，是否涉及后端 API contract 变更？"
  - 选项：`"是，后端已改/在改（给个 PR 链接）"` / `"不涉及后端，纯客户端调整"` / `"不确定，要去问后端"`

> **场景**：改动跨端但只动了一端。
- "diff 显示只动了 androidMain，没动 iosMain —— iOS 也要跟着改吗？"
  - 选项：`"iOS 已经有了/不需要"` / `"iOS 跟进，单独 PR"` / `"忘了 iOS，需要补"`

> **场景**：可能涉及关联仓库。
- "这个改动看起来跟 web 端 / admin 后台有联动 —— 那些 repo 需要拉到工作区一起对照吗？"

---

### 1️⃣ 意图不明

> **场景**：一个 commit 同时改了看起来不相关的两块。
- "这里同时改了 A 模块和 B 模块，是同一个目的还是顺手改的？"
  - 选项：`"同一目的（要 review 联动正确性）"` / `"顺手改的（应该拆 commit）"` / `"忘了，记不清了"`

> **场景**：函数被重命名/重构。
- "这个函数从 X 改名为 Y，是为了语义更准、还是兼容某个外部约定（API spec / spec 文档）？"

> **场景**：增加了配置项 / 开关。
- "新增的 `enableXxx` flag，默认值打算设成 true 还是 false？灰度策略是什么？"

---

### 2️⃣ 实现路径选择

> **场景**：用了 `Flow` 而场景看起来像一次性请求。
- "这块改用 Flow 而非 suspend fun，是要持续观察数据变化、还是单次返回？"
  - 选项：`"持续观察（Flow 是对的）"` / `"单次返回（应该用 suspend）"` / `"两者都需要"`

> **场景**：网络请求没加超时 / 没加重试。
- "Ktor 调用没显式 `timeout` 块，是认为全局配置够用、还是疏漏？"
  - 选项：`"全局配置够用（Recommended）"` / `"疏漏，应该加"` / `"业务上希望一直等"`

> **场景**：用了 `runCatching` 包住协程。
- "`runCatching` 包住 suspend 调用，CancellationException 会被吞 —— 这是有意的吗？"
  - 选项：`"无意，要修"` / `"有意，不希望取消传播"` / `"不知道这个坑（请展开解释）"`

> **场景**：异常处理选择。
- "这里 try-catch 后 log + 返回 null，希望上游怎么处理这个 null？"
  - 选项：`"当作正常空数据"` / `"当作失败要重试"` / `"我也没想清楚"`

---

### 3️⃣ 边界假设

> **场景**：用了 `!!` 非空断言。
- "`xxx!!` 这里能确认调用上下文一定非空吗？还是边界没考虑全？"

> **场景**：集合操作没处理空。
- "这个集合 `.first()` 调用，能保证上游一定非空吗？"
  - 选项：`"保证非空（业务前提）"` / `"应该用 firstOrNull"` / `"取决于网络，应该兜底"`

> **场景**：可空链滥用。
- "`a?.b?.c?.d` 这一长串可空链，链路上每一段都可能为 null 是预期吗？"

---

### 4️⃣ 性能与并发

> **场景**：`LaunchedEffect(Unit)` 包了长任务。
- "`LaunchedEffect(Unit)` 跟随首次组合启动这个长任务 —— 是要首次进页面就跑，还是其实要响应某个事件？"
  - 选项：`"首次进页面跑（Recommended，但要看是否需要 cancel-aware）"` / `"应该响应事件"` / `"应该挪到 ViewModel"`

> **场景**：循环里有 IO / suspend 调用。
- "这个循环里串行调用 N 次网络，预期 N 大概多少？要不要换成并发？"
  - 选项：`"N 通常很小（< 5）"` / `"N 可能很大（应该并发或分页）"` / `"N 不确定（要做压测）"`

> **场景**：PowerSync watch / 大查询。
- "PowerSync watch 这个查询，预期数据量级是多少行？需不需要 LIMIT 或分页？"

> **场景**：在 composable 里创建 Flow / 集合。
- "这个 `flow { ... }` 在 composable 里创建，每次重组都会新建，应该 `remember` 包一下吗？"
  - 选项：`"应该 remember"` / `"故意每次新建"` / `"应该挪到 ViewModel"`

---

### 5️⃣ Compose 特定

> **场景**：`remember` 缺 key。
- "`remember { ... }` 这里没加 key，但内部依赖了 `xxx`，xxx 变了应该重建吗？"

> **场景**：`LaunchedEffect` key 不完整。
- "`LaunchedEffect(a)` 内部还用到了 b，b 变了不重启是有意的吗？"

> **场景**：`mutableStateOf` 在 composable 外被持有。
- "这个 `mutableStateOf` 放在 object/ViewModel 里被多处 composable 共享 —— 是有意的共享状态，还是应该走 Flow？"

---

### 6️⃣ 平台差异

> **场景**：改了 expect / actual。
- "expect 改了签名，所有 actual 都跟着改了吗？iOS 和 Android 的语义保持一致吗？"
  - 选项：`"都改了，语义一致"` / `"androidMain 改了，iosMain 还没"` / `"语义不需要严格一致（解释下）"`

> **场景**：只在 androidMain 加了功能。
- "Android 用 Firebase 上报这个事件，iOS 这边没对应实现 —— 是 Android-only 功能、还是计划后补 iOS？"

> **场景**：权限相关。
- "这个权限 Android 走 runtime permission，iOS 走 Info.plist —— 两边都改了吗？两端用户体验对齐了吗？"

> **场景**：协程上下文跟 GCD 桥接。
- "iOS actual 这里手动切到了 main queue，Android actual 用 `Dispatchers.Main` —— 行为差异（特别是 Main.immediate）能接受吗？"

---

### 7️⃣ 第三方依赖升级

> **场景**：动了 `libs.versions.toml`。
- "这次升了 Ktor / PowerSync / Sketch 版本 —— 已知有 breaking change 吗？看过 changelog 了吗？"
  - 选项：`"看了，没影响"` / `"看了，有迁移要做（已做）"` / `"没看，靠测试覆盖"`

---

### 8️⃣ 数据 / 同步语义

> **场景**：PowerSync schema 改动。
- "PowerSync schema 这次改了 X 表，有迁移逻辑吗？老数据怎么处理？"

> **场景**：序列化兼容性。
- "新增字段 `xxx`，老版本 App 收到带 xxx 的 payload 会怎样？"
  - 选项：`"忽略未知字段（@Serializable 默认行为，OK）"` / `"加了 @SerialName 别名"` / `"应该考虑兼容性，但还没"`

---

## 🚫 不要这样问

- ❌ "看起来 OK 吗？"（太开放）
- ❌ "你觉得这里应该怎么改？"（review 不是把判断推给用户）
- ❌ 一次问 10 个问题（用户会答不下去）
- ❌ 问纯文档型问题（"Ktor 怎么用？"——这应该自己搜，不是问开发者）

---

## ✅ 好提问 vs 差提问

| ❌ 差 | ✅ 好 |
|---|---|
| "这个看起来有问题？" | "这个 `runCatching` 会吞掉 CancellationException，是有意的吗？" |
| "你为什么这样写？" | "改用 Flow 是要持续观察、还是单次结果？（给出 2-3 选项）" |
| "可以解释一下吗？" | "这个 `!!` 调用上下文能保证非空吗？还是边界没考虑？" |

---

## 📝 问完之后

每轮对话的 Q&A 都要记录到最终报告的「用户对话记录」一节。
不要让澄清过的内容在报告里又当成「待确认」列出 —— 用户回答了就是定论。
